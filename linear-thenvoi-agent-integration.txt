## TL;DR

Build a **Linear bridge agent** -- an OAuth application that appears as a delegatable AI agent inside Linear workspaces. When a user tags the Thenvoi agent on a Linear issue, a webhook handler acknowledges immediately, then triggers the bridge agent. The bridge agent creates a Thenvoi chat room, adds coding and reviewer agents as participants, and monitors the conversation -- relaying progress to Linear as activities. Agents coordinate naturally in the chat room using existing platform messaging. No custom pipeline engine needed; **the chat room is the orchestration**.

The end-state demo: a user comments on a Linear ticket (e.g., "implement dark mode"), tags `@Thenvoi`, and a coding agent develops the solution while a reviewer agent reviews it -- all coordinated in a Thenvoi chat room, with progress and PR links posted back to Linear automatically.

---

## Why This Matters

### Problem

Today, each coding agent (Codex, Claude Code, Devin, Copilot, Cursor) has its own Linear integration. There is **no universal routing layer** between project management and coding agents. Users must:

- Manually choose which agent to delegate to before knowing which is best for the task
- Context-switch between agent dashboards for status updates
- Manually hand off between agents (e.g., code with one, review with another)
- Set up and maintain separate integrations for each agent

### Why Now

- Linear launched its [Agent Developer Platform](https://linear.app/developers/agents) (May 2025) -- agents are now first-class workspace members
- No dominant solution exists for Linear-to-coding-agent routing (unlike Jira which has Rovo Dev)
- Thenvoi already has native adapters for most target AI frameworks
- This is the most compelling demo of Thenvoi's multi-agent orchestration value

### Business Value

| Value                  | Description                                                                                   |
| ---------------------- | --------------------------------------------------------------------------------------------- |
| **Dogfooding**         | Uses Thenvoi's own platform for internal development workflow                                 |
| **Showcase use case**  | "Linear issue -> multi-agent chat room -> PR" is the best demo of agent-to-agent communication |
| **Market positioning** | First universal routing layer for Linear's agent ecosystem                                    |
| **Revenue potential**  | Linear has massive developer adoption; coding agent routing is a high-value workflow          |

### Detailed User Flow

**Step 1: Trigger**
- User opens a Linear issue (e.g., INT-200: "Implement dark mode")
- User comments `@Thenvoi implement this` or delegates the issue to the Thenvoi agent
- Linear fires an `AgentSessionEvent` webhook to Thenvoi

**Step 2: Acknowledge**
- Webhook handler receives the event, verifies HMAC signature
- Synchronously posts a `thought` activity within 10 seconds: "Analyzing issue INT-200... setting up coding workflow"
- Triggers the Linear bridge agent asynchronously with issue context

**Step 3: Set Up Chat Room**
- Bridge agent parses issue context: title, description, labels, comments, `promptContext` (workspace/team guidance)
- Creates a Thenvoi chat room with the issue details as the initial message
- Adds coding agent (e.g., Claude Code) and reviewer agent (e.g., Codex) as participants
- Which agents to add is determined by the bridge agent's LLM reasoning based on issue context and its system prompt

**Step 4: Develop (Claude Code)**
- Claude Code receives the task via the Thenvoi chat room -- it's just a normal Thenvoi agent
- Reads the repo, understands the codebase structure
- Implements the feature (dark mode CSS variables, theme toggle, component updates)
- Creates a branch, commits, pushes, opens a draft PR
- Bridge agent monitors the chat room and relays progress to Linear as `action` activities

**Step 5: Review (Codex)**
- Codex receives the PR URL and diff via the chat room conversation
- Reviews code quality, edge cases, accessibility, naming conventions
- Posts review comments on the GitHub PR
- If changes are needed: sends feedback to Claude Code in the chat room; Claude Code iterates and pushes fixes
- If approved: signals completion in the chat room

**Step 6: Finalize**
- Bridge agent detects reviewer approval in the chat room
- Posts a `response` activity to Linear with the final summary and PR link
- Updates the agent session with the external PR URL
- Session transitions to `complete`

### Clarification Flow

At any point, if the bridge agent determines user input is needed:

```
Bridge agent calls linear_post_elicitation → Linear shows prompt to user
                                           → User responds in comment
                                           → Linear fires "prompted" webhook
                                           → Webhook handler routes to bridge agent
                                           → Bridge agent relays to chat room
                                           → Agents continue with clarification
```

## Architecture

### Design Principle: Use the Platform, Don't Rebuild It

Thenvoi is already a multi-agent orchestration platform. The Linear integration should plug into it as a bridge, not build a parallel orchestration system on top. Agents coordinate in a chat room -- the existing platform feature. The bridge agent's LLM reasoning replaces rigid pipeline definitions. Coding agents are just normal Thenvoi agents that know nothing about Linear.

### Component 1: Webhook Handler

Stateless HTTP endpoint that receives Linear webhooks and triggers the bridge agent via the existing Thenvoi API.

| Property | Value |
|----------|-------|
| Auth | HMAC-SHA256 signature verification (Linear's webhook signing) |
| Response SLA | Acknowledge within 10 seconds |
| Behavior | Verify signature, post immediate `thought` activity to Linear, call Thenvoi API to message bridge agent |

**How it works:** The webhook handler doesn't require new Thenvoi infrastructure. It receives Linear's webhook, then calls the existing Thenvoi Chat API (`POST /chats/{id}/messages`) to send a message to the bridge agent with the issue context. The bridge agent is already listening in a dedicated chat room.

**Implementation options:**
1. **Serverless function** - AWS Lambda, Cloudflare Worker, or Vercel function (simplest, recommended for MVP)
2. **Standalone service** - Small service deployed alongside Thenvoi
3. **Thenvoi API extension** - Add a `/webhooks/inbound/linear` route (more coupling, less recommended - maybe for future implementation)

### Component 2: Linear Bridge Agent

An LLM-powered Thenvoi agent with two sets of tools: Linear tools (for posting activities back to Linear) and platform tools (for creating chat rooms and coordinating agents). It doesn't code -- it coordinates.

**Linear tools** (thin wrappers around Linear's GraphQL API):

| Tool | GraphQL Operation | Purpose |
|------|-------------------|---------|
| `linear_post_thought` | `agentActivityCreate` (type: thought) | Post status updates to Linear |
| `linear_post_action` | `agentActivityCreate` (type: action) | Post work progress to Linear |
| `linear_post_elicitation` | `agentActivityCreate` (type: elicitation) | Ask user for clarification |
| `linear_complete_session` | `agentSessionUpdate` | Mark session complete, attach PR URL |
| `linear_post_error` | `agentActivityCreate` (type: error) | Report failures to Linear |

**Platform tools** (existing Thenvoi SDK):

| Tool | Purpose |
|------|---------|
| `create_chatroom` | Create a conversation for agents to coordinate in |
| `add_participant` | Add coding/reviewer agents to the chat room |
| `send_message` | Send messages to agents in the chat room |
| `lookup_peers` | Discover available agents and their capabilities |

**The LLM is the orchestrator.** It reads the chat room conversation and decides when to post updates to Linear, when to nudge agents, when to escalate to the user via `elicitation`. No rigid pipeline definitions needed -- behavior is governed by the bridge agent's system prompt.

### Linear OAuth App

Registered on Linear with `actor=app`. Appears as a delegatable agent in workspaces.

| Property | Value |
|----------|-------|
| OAuth mode | `actor=app` with client credentials grant |
| Scopes | `app:assignable`, `app:mentionable`, `read`, `write`, `issues:create`, `comments:create` |
| Delegation model | Human stays assignee, Thenvoi agent is delegate (per Linear's API design) |

## Linear Activity Mapping

The bridge agent maps chat room events to Linear's activity types:

| Chat Room Event | Linear Activity Type | Example |
|-----------------|---------------------|---------|
| Bridge agent starts | `thought` | "Analyzing issue INT-200..." |
| Coding agent reads files | `action` | action: "Read file", parameter: "src/styles/theme.css" |
| Coding agent edits code | `action` | action: "Edit file", parameter: "src/components/ThemeToggle.tsx" |
| Coding agent creates PR | `action` + external URL | action: "Create PR", result: "PR #42 opened" |
| Bridge agent needs input | `elicitation` | "Which color palette should I use for dark mode?" |
| Workflow completes | `response` | "Done. PR #42 implements dark mode with CSS variables..." |
| Agent encounters error | `error` | "Failed to push to repository: permission denied" |

**Agent plans** (visible in Linear UI):
```json
[
  { "content": "Analyze codebase and plan implementation", "status": "completed" },
  { "content": "Implement dark mode CSS variables", "status": "completed" },
  { "content": "Add theme toggle component", "status": "inProgress" },
  { "content": "Code review", "status": "pending" },
  { "content": "Finalize and open PR", "status": "pending" }
]
```

## Sandbox & Environment Provisioning

Coding agents need a sandboxed environment with repo access to do their work. This is the hardest infrastructure problem in the integration.

| Concern                | Approach                                                                                         |
| ---------------------- | ------------------------------------------------------------------------------------------------ |
| **Where agents run**   | Each coding agent session gets an isolated sandbox (container or VM) with the target repo cloned |
| **Repo access**        | GitHub App installation token scoped to the target repo, injected into sandbox                   |
| **Branch isolation**   | Each task creates a feature branch; agents never push to main                                    |
| **Cleanup**            | Sandbox destroyed after session completes or times out                                           |
| **MVP simplification** | For internal dogfooding, agents run against repos we control with pre-configured access          |

**Deferred decision**: The exact sandbox provider (Fly Machines, E2B, Modal, or existing infra) will be decided in a separate ticket focused on the demo environment. This decision depends on:
- **Latency**: Cold start time for spinning up sandboxes
- **Cost**: Per-session compute costs at expected volume
- **Security**: Isolation guarantees and secret injection mechanisms
- **Developer experience**: How easy it is to debug and iterate

For the internal dogfooding demo, we may bypass this entirely by running agents against local dev environments or pre-provisioned VMs.

---

## Git Provider Dependency

The entire flow assumes **GitHub** as the git provider:

- PR creation via `gh` CLI or GitHub API
- Review comments posted to GitHub PRs
- CI status checked via GitHub Checks API
- Webhook notifications from GitHub for CI results

**Explicitly out of scope**: GitLab, Bitbucket, Azure DevOps. The architecture supports other providers in the future by swapping git-related tools on the coding agents, but MVP is GitHub-only.

## Failure & Recovery

| Failure Scenario | Detection | Recovery |
|-----------------|-----------|----------|
| **Bridge agent crashes** | Webhook handler health check / heartbeat timeout | Restart bridge agent with chat room context; post `error` activity to Linear |
| **Coding agent times out** | Bridge agent monitors for inactivity (configurable, default: 30 min) | Post timeout `error` to Linear; complete session with partial results |
| **Coding agent produces bad code** | Reviewer agent rejects PR | Bridge agent relays feedback; coding agent iterates (max 3 rounds) |
| **All iterations exhausted** | Bridge agent counts review rounds | Post summary of attempts to Linear; complete session with "needs human review" |
| **Webhook delivery fails** | Linear retries (built-in) | Idempotency key on webhook handler prevents duplicate processing |
| **Linear API unavailable** | HTTP errors from Linear tools | Bridge agent retries with backoff; buffers activities locally |

## Observability

| Layer | What to Log/Trace | Tool |
|-------|-------------------|------|
| **Webhook handler** | Request received, HMAC verification, bridge agent trigger | Structured logs + request tracing |
| **Bridge agent** | Tool calls (Linear + platform), LLM reasoning, chat room events | Thenvoi platform telemetry |
| **Chat room** | All messages between agents, timestamps, token usage | Platform message log |
| **Linear activities** | Every activity posted, response codes, latency | Linear tool call logs |
| **Cost tracking** | Per-session token usage by agent, total cost | Platform billing metrics |

Key metrics to monitor:
- **Time to first activity**: webhook received → first `thought` posted (target: < 10 sec)
- **End-to-end latency**: webhook → session complete
- **Success rate**: sessions that produce a PR vs. sessions that error
- **Cost per session**: total LLM tokens across all agents
- **Review iteration count**: how many develop-review cycles per task

## Security

| Concern | Mitigation |
|---------|-----------|
| Webhook authenticity | HMAC-SHA256 signature verification |
| Agent credentials | Stored in Thenvoi vault, never in Linear |
| Repository access | Scoped per-agent via GitHub App installation tokens |
| Cost control | Per-task cost limits, timeout guards, max review iterations |
| Data leakage | Issue content stays within Thenvoi boundary |
| Rate limiting | Respect Linear's 250,000 points/hour limit |
| Sandbox isolation | Coding agents run in ephemeral containers with no access to other repos |

## Phased Delivery

### Phase 1: Linear Bridge Infrastructure

Build the generic Linear-to-Thenvoi bridge -- the reusable infrastructure that connects Linear's agent platform to Thenvoi's multi-agent coordination.

| Component | Scope |
|-----------|-------|
| Linear OAuth App | Register `actor=app` with required scopes; appears as delegatable agent in Linear |
| Webhook handler | Serverless function: receive `AgentSessionEvent`, verify HMAC, post immediate `thought`, call Thenvoi API |
| Linear tools | 5-6 tools wrapping Linear GraphQL (`agentActivityCreate`, `agentSessionUpdate`, `commentCreate`) |
| Bridge agent | System prompt + Linear tools + platform tools; generic orchestrator that coordinates agents based on task |

**Outcome**: A working bridge that can receive Linear delegations and coordinate agents in Thenvoi. No specific coding workflow yet -- just the infrastructure.

### Phase 2: Dogfooding Demo (Coding Workflow)

Build the specific coding workflow demo on top of Phase 1 infrastructure.

| Component | Scope |
|-----------|-------|
| Coding agents | Claude Code (developer) + Codex (reviewer) connected to the bridge |
| Sandbox environment | Decision on provider (Fly, E2B, Modal, or local) -- separate ticket |
| Git integration | GitHub App for repo access, branch creation, PR workflow |
| Demo scenario | "Implement dark mode" end-to-end on internal repos |

**Outcome**: The target demo -- user tags `@Thenvoi` on a Linear issue, coding agent implements, reviewer approves, PR link posted back.

### Phase 3: External Beta (optional - according to usage)

| Enhancement | Description |
|-------------|-------------|
| Multi-workspace support | Per-workspace config for which agents to use |
| More agents | Add Devin, other coding agents as adapters mature |
| Workspace `promptContext` | Respect Linear's workspace/team guidance in bridge agent context |
| Cost tracking & limits | Per-workspace billing, cost caps per session |

### Phase 4: GA (optional - according to usage)

| Enhancement | Description |
|-------------|-------------|
| Analytics dashboard | Success rates, cost per task, agent performance comparison |
| Auto-triage | Analyze issue complexity and route to appropriate agent mix |
| Custom agent configs | Workspace owners configure preferred agents, review policies |
| Additional git providers | GitLab, Bitbucket support |

## Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Linear Agent APIs are in **Developer Preview** | API changes could break integration | Thin abstraction via Linear tools; monitor changelog |
| 10-second response requirement | Webhook handler must be fast | Synchronous acknowledgment before async bridge agent |
| Coding agent costs can be high | Budget overruns per task | Cost limits per task; user approval for expensive ops |
| Review loops could run indefinitely | Wasted compute, no result | Max iteration limit (default: 3) |
| Agent quality varies by task type | Poor results reflect on Thenvoi | Start with internal dogfooding; iterate on system prompt |
| Sandbox provisioning latency | Slow startup degrades UX | Pre-warm containers; post progress activities during setup |

## Out of Scope

- Auto-triage rules (requires Linear Business/Enterprise plan -- Phase 3)
- Analytics dashboard for agent performance metrics (Phase 3)
- Cursor and Copilot adapters (limited API availability -- add when APIs mature)
- Cost billing to workspace owners (internal dogfooding first)
- Open-sourcing the Linear agent (evaluate after internal validation)
- Non-GitHub git providers (Phase 3)

## Success Criteria

The integration is successful when we can demonstrate:

1. User tags `@Thenvoi` on a Linear issue
2. Webhook acknowledges within 10 seconds with a `thought` activity
3. Bridge agent creates a Thenvoi chat room and adds coding + reviewer agents
4. Claude Code implements the requested change and opens a PR
5. Codex reviews the PR and requests changes or approves
6. If changes requested, Claude Code iterates (up to 3 rounds)
7. Final PR link and summary are posted back to the Linear issue
8. Entire flow is visible as a timeline of activities in Linear
9. No manual intervention required after the initial tag
